Помимо основных сущностей предметной области (КРД и связанные данные), информационная модель включает таблицы для управления процессами отчётности, персонализации интерфейса и контроля безопасности. Данные таблицы обеспечивают гибкость настройки системы под нужды конкретного сотрудника и соответствие требованиям аудита.

Таблица 2.14 – Структура таблицы krd.report_templates
| Поле | Тип | Ограничения | Описание |
| --- | --- | --- | --- |
| id | SERIAL | PK | Уникальный ID шаблона |
| name | VARCHAR(255) | NOT NULL | Название шаблона (например, «Полная выписка») |
| config_json | JSONB | NOT NULL | JSON-конфигурация выбранных секций и полей |
| template_type | VARCHAR(50) | DEFAULT 'excel' | Тип экспорта (excel) |
| usage_count | INTEGER | DEFAULT 0 | Счётчик частоты использования шаблона |
| is_default | BOOLEAN | DEFAULT FALSE | Флаг шаблона по умолчанию |
| is_deleted | BOOLEAN | DEFAULT FALSE | Мягкое удаление |

Таблица `krd.report_templates` хранит пользовательские конфигурации для массового экспорта данных. Ключевым элементом является поле `config_json` (тип JSONB), которое сохраняет структуру отчёта: список включенных секций (социальные данные, адреса, эпизоды СОЧ) и конкретных полей внутри этих секций. Это позволяет пользователю один раз настроить формат выгрузки и переиспользовать его без повторного выбора полей.

Таблица 2.15 – Структура таблицы krd.generated_documents
| Поле | Тип | Ограничения | Описание |
| --- | --- | --- | --- |
| id | SERIAL | PK | Уникальный ID записи |
| krd_id | INTEGER | FK → krd.id | Ссылка на карточку розыска |
| template_id | INTEGER | FK → report_templates | Использованный шаблон (если применимо) |
| document_data | BYTEA | NOT NULL | Бинарные данные файла (архив) |
| file_name | VARCHAR(255) | NOT NULL | Исходное имя файла |
| generated_at | TIMESTAMP | DEFAULT NOW() | Дата и время генерации |
| generated_by | INTEGER | FK → users | ID пользователя, создавшего документ |
| is_deleted | BOOLEAN | DEFAULT FALSE | Мягкое удаление |

Таблица `krd.generated_documents` реализует функцию архивации созданных документов. В отличие от шаблонов, здесь хранятся физические файлы (DOCX, PDF) в бинарном виде (BYTEA). Это гарантирует, что история документооборота сохраняется независимо от файловой системы рабочей станции и доступна для аудита.

Таблица 2.16 – Структура таблицы krd.user_settings
| Поле | Тип | Ограничения | Описание |
| --- | --- | --- | --- |
| id | SERIAL | PK | Уникальный ID настроек |
| user_id | INTEGER | FK → users.id, UNIQUE | Привязка к пользователю |
| config_json | JSONB | NOT NULL | JSON с параметрами UI (шрифт, темы) |
| updated_at | TIMESTAMP | DEFAULT NOW() | Дата последнего изменения |

Таблица `krd.user_settings` отвечает за персонализацию рабочего места оператора. Поле `config_json` хранит сериализованные настройки интерфейса: предпочитаемый размер шрифта, состояние свёрнутых панелей и пользовательские цветовые схемы.

Таблица 2.17 – Структура таблицы krd.user_themes
| Поле | Тип | Ограничения | Описание |
| --- | --- | --- | --- |
| id | SERIAL | PK | Уникальный ID темы |
| user_id | INTEGER | FK → users.id | Владелец темы |
| theme_name | VARCHAR(100) | NOT NULL | Название темы |
| config_json | JSONB | NOT NULL | Цветовая схема (HEX коды) |
| is_active | BOOLEAN | DEFAULT FALSE | Активная тема сейчас |
| is_default | BOOLEAN | DEFAULT FALSE | Базовая тема системы |

Таблица `krd.user_themes` позволяет создавать неограниченное количество визуальных тем оформления. Конфигурация цветов хранится в JSONB, что дает возможность гибко менять цвета кнопок, фона и текста без модификации кода приложения.

Таблица 2.18 – Структура таблицы krd.user_sessions
| Поле | Тип | Ограничения | Описание |
| --- | --- | --- | --- |
| id | SERIAL | PK | Уникальный ID сессии |
| user_id | INTEGER | FK → users.id | Владелец сессии |
| login_time | TIMESTAMP | NOT NULL | Время входа |
| logout_time | TIMESTAMP | NULL | Время выхода |
| is_active | BOOLEAN | DEFAULT TRUE | Статус сессии |

Таблица `krd.user_sessions` используется для контроля безопасности. Она фиксирует каждое успешное вхождение в систему. Администратор может использовать эти данные для выявления аномальной активности (например, вход в нерабочее время) или для подсчета реального времени работы сотрудников в системе.

2.4.4 Выводы по проектированию базы данных
1. Разработана полноценная реляционная схема `krd`, состоящая из 26 таблиц. Структура покрывает все требования ТЗ: от учёта персональных данных до гибкой настройки отчётности и интерфейса.
2. Активное использование типа данных `JSONB` (в таблицах `field_mappings`, `report_templates`, `user_settings`) позволило реализовать гибкие конфигурации (маппинг шаблонов, настройки отчётов, темы оформления) без изменения структуры БД (DDL-операций) при обновлении функционала.
3. Реализован механизм мягкого удаления (`is_deleted`) для всех критичных таблиц. Это обеспечивает сохранность истории операций и возможность восстановления ошибочно удалённых записей администратором.
4. Бинарные данные (фотографии, сканы документов, сгенерированные файлы) хранятся в полях типа `BYTEA`. Это упрощает процедуру резервного копирования (достаточно дампа базы данных) и гарантирует атомарность хранения файла вместе с его метаданными.
5. Спроектирована система аудита (`audit_log`) и контроля сессий (`user_sessions`), обеспечивающая соответствие требованиям информационной безопасности и подотчётности действий пользователей в изолированном контуре.
